## Context

Use when designing core mechanics, tuning difficulty, or diagnosing why a feature feels "off." Applies to any project. Provides a mental checklist before writing design docs or reviewing another designer's work.

## Core Patterns

### MDA Framework: Mechanics → Dynamics → Aesthetics
**Mechanics** (what you build) → **Dynamics** (what emerges) → **Aesthetics** (what player feels). Design mechanics, playtest reveals dynamics, dynamics create aesthetics. Example: Celeste's dash cooldown → chain dash-jump-dash → "I am masterful."

### Core Loop: 30s, 5min, 30min
- **30s:** Core interaction (button → feedback → next button)
- **5min:** Complete challenge (clear level, fight encounter)
- **30min:** Full session (3-5 levels, natural stop)

### Feedback Loops
- **Positive:** Success → more success (momentum). Risk: runaway winner.
- **Negative:** Success → pulled back (fairness). Risk: feels manipulative.
- **Combine:** Local snowball + global equilibrium.

### Risk/Reward
Every decision needs different costs and rewards. Safe path vs risky path. If one dominates, choice disappears.

## Key Examples

### Flow State
Challenge matches skill. Too hard = anxiety. Too easy = boredom. As player improves, introduce new challenges.

### Intrinsic vs Extrinsic Motivation
- **Extrinsic:** Play for rewards. Fast but shallow.
- **Intrinsic:** Play because satisfying. Slow but deep.
- **Mix:** Early (extrinsic), mid (intrinsic matures), late (intrinsic anchor).

### Loss Aversion
Humans feel loss 2× more intensely than gains. Make loss meaningful but fair. Avoid permanent loss early game.

## Anti-Patterns

- **Design by Committee:** No vision owner. Fix: One designer owns direction.
- **Feature Creep:** Mechanics don't serve core loop. Fix: Every feature serves core. Say no 10× per yes.
- **80% Trap:** Polish non-critical systems. Fix: Critical path gets 80% time.
- **Balance First, Fun Later:** Fix: Prototype → playtest → polish fun → then balance.
